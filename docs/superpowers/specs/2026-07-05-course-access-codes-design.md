# נעילת הקורס: קודי גישה אישיים + Waitlist

תאריך: 2026-07-05 · סטטוס: מאושר על ידי חניתה (בשיחה)

## מטרה

לסגור את הגישה החופשית ל-course.octaloom.com עד שיוחלט על מחיר וסליקה, בלי לפגוע ב-14 הנרשמים הקיימים. גישה חדשה ניתנת רק בקוד אישי חד-פעמי שחניתה מייצרת. מבקרים בלי קוד משאירים מייל ב-waitlist.

## החלטות מפתח

- **בלי Clerk Restricted mode.** ההרשמה נשארת פתוחה טכנית, כדי שמחזיקי קוד יוכלו להירשם. הנעילה היא על התוכן, בצד השרת.
- **קוד לא קשור מראש למייל.** הוא ננעל לחשבון שמימש אותו ראשון (עם אופציה בדף האדמין לנעול קוד למייל ספציפי מראש).
- **מייל התודה של MailerLite עובר לרגע מימוש הקוד** (לא רגע ההרשמה), כדי שלא יישלח "הגישה שלך פתוחה" למי שאין לו גישה.
- **מוכן למשפך עתידי:** webhook של סליקה עתידית יקרא לאותו API של יצירת קוד. שום דבר לא נזרק.

## ארכיטקטורה

### אחסון: Upstash Redis (דרך Vercel Marketplace)
- `code:<CODE>` → JSON: `{ status: "active"|"redeemed", note, lockedEmail?, createdAt, redeemedBy?, redeemedByEmail?, redeemedAt? }`
- `codes:index` → set של כל הקודים (לרשימה בדף האדמין).
- env: `KV_REST_API_URL`, `KV_REST_API_TOKEN` (או שמות Upstash המקבילים).

### מודל גישה: Clerk publicMetadata
- `publicMetadata.courseAccess: true` = רואה קורס וכלים.
- Backfill חד-פעמי: `scripts/grant-access-existing.mjs` מסמן את כל משתמשי ה-production הקיימים (מפתח מ-.env.prod.local).

### עמודים ו-routes
| נתיב | תפקיד |
|---|---|
| `/sign-up` (קיים, משתנה) | מסך waitlist מעוצב: "הקורס נפתח בקרוב" + טופס מייל + קישור קטן "יש לך קוד גישה? להרשמה" שמוביל לטופס Clerk האמיתי |
| `/redeem` (חדש) | למשתמש מחובר בלי גישה: שדה קוד + הסבר. מימוש תקף → גישה + מעבר לקורס |
| `/admin/codes` (חדש) | רק לחניתה (לפי `ADMIN_EMAILS` env): יצירת קודים (כמות, הערה, נעילת-מייל אופציונלית), רשימה עם סטטוס והעתקה |
| `/api/waitlist` (חדש, public) | מוסיף מייל ל-group "Course Waitlist" ב-MailerLite |
| `/api/redeem` (חדש) | ולידציה אטומית ב-Redis, סימון redeemed, `courseAccess=true`, הוספה ל-group "נרשמי קורס לינקדאין" (מפעיל את סיקוונס התודה) |
| `/api/admin/codes` (חדש) | GET רשימה / POST יצירה, מוגן לפי ADMIN_EMAILS |
| `/api/clerk-webhook` (קיים, משתנה) | user.created כבר לא מוסיף ל-group הקורס (עובר ל-redeem). נשאר לצרכי לוג/עתיד |

### אכיפת הגישה
בדיקת שרת משותפת (helper `requireCourseAccess`) בעמודי התוכן (`/`, `/course/[id]`, `/tools/*`): משתמש מחובר בלי `courseAccess` → redirect ל-`/redeem`. לא מחובר → התנהגות קיימת (sign-in).

### פורמט קוד
`LNKD-XXXX` (4 תווים אקראיים, בלי תווים דו-משמעיים כמו O/0). לא רגיש לרישיות.

## מקרי קצה
- קוד מומש פעמיים → שגיאה ידידותית "הקוד כבר מומש".
- קוד נעול-מייל שמנסה לממש חשבון אחר → שגיאה עם הנחיה לפנות לחניתה.
- Rate limit על `/api/redeem` ו-`/api/waitlist` (שימוש חוזר ב-lib/ratelimit הקיים).
- משתמש עם גישה שמגיע ל-`/redeem` → redirect הביתה.
- מימוש: פעולה אטומית (GETSET/WATCH) כדי שאותו קוד לא ימומש פעמיים במקביל.

## מחוץ לסקופ (שלב עתידי)
- אינטגרציית סליקה (תיצור קודים דרך אותו API).
- שליחת קוד אוטומטית במייל.
- מיילי waitlist שיווקיים (הרשימה נבנית, השליחה ידנית מ-MailerLite).

## בדיקות קבלה
1. משתמש קיים (מה-14) נכנס ורואה הכל, בלי לעבור redeem.
2. מבקר חדש בלי קוד: רואה waitlist, משאיר מייל, מופיע ב-group Course Waitlist.
3. נרשם חדש עם קוד תקף: נרשם → redeem → רואה קורס → מופיע ב-group נרשמי קורס → מקבל מייל תודה.
4. אותו קוד בפעם שנייה → נדחה.
5. חניתה רואה את `/admin/codes`; משתמש אחר מקבל 404/redirect.
